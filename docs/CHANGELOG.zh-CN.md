# 更新日志

Youwee 的所有重要更改都将记录在此文件中。

格式基于 [Keep a Changelog](https://keepachangelog.com/en/1.1.0/)，
本项目遵循 [语义化版本](https://semver.org/spec/v2.0.0.html)。

## [Unreleased]

### 修复
- **抓取下载 Rust 传输接管** - 直链媒体抓取现在会优先调用已编译的 `crawler_downloader` Rust binary 执行实际文件传输，Python 继续负责扫描、过滤和报告；只有 Rust 路径不可用或失败时才回退到内置下载器
- **抓取 host 级连接门控** - 探测请求、分段下载和单流 fallback 现在统一共用同一套 host 级并发槽位，启动时会打印各 host 上限，图床被打满时也会输出等待日志
- **抓取分段下载 worker 调度** - Range 下载现在会在拆分 pending chunk 之后再计算真实 worker 数，记录实际 dispatch 并发，并在长时间等待时输出心跳日志，避免大 chunk 仍在传输时看起来像假死
- **更激进的自适应分段下载** - 抓取直链媒体现在会按文件大小和域名画像动态提升到最高 12 段并发，未完成的 `.part` 子段会续传而不是从 0 重下，失败任务也会保留子段状态供下次恢复，同时连接池也扩大到能承受高分段并发
- **分段下载子段独立重试** - 并行 `Range` 下载现在会对每个子段单独重试，而不是某一段失败就整文件重下；Google 相册图片和视频直链也都纳入同一套分段链路
- **爬虫域名级读取超时策略** - 爬虫现在会对 `i.ibb.co`、Google 用户内容这类慢图床单独放宽读取超时，同时保持默认连接超时较短，并在启动日志里打印当前生效的超时策略
- **爬虫图床下载稳定性** - 图片下载现在会使用每张图自己的来源页面作为 `Referer`，启动时打印真实下载超时配置，并放宽 `stable_large` 模板过于保守的限速默认值，避免大相册任务被自身参数压慢
- **爬虫下载超时与重试调优** - 直链图片下载现在拆分连接超时和读取超时、增大流式分块，并对不可重试的 HTTP 失败停止重复重试，减少慢图床上的超时和无效等待
- **爬虫扫描完成后未进入下载阶段** - 修复 `crawl_pages()` 返回值数量不匹配（`expected 5, got 4`）导致扫描成功结束后在真正开始下载前直接中断的问题
- **爬虫输出目录解析与报告路径信息** - 完成弹窗和打开目录动作现在会优先依据预览/报告产物与保存路径字段定位真实媒体目录，报告本身也补齐了足够的路径信息，避免再退回到重试目录或上级输出目录
- **爬虫 sidecar UTF-8 日志输出** - sidecar 启动 `image_crawler.py` 前现在会强制设置 `PYTHONIOENCODING=utf-8`、`PYTHONUTF8=1` 和无缓冲输出，避免应用里的中文任务日志和目录路径出现乱码

### 新增
- **页面解析流新增隐藏的 JPG 序列播放路径** - 应用内 HLS 播放器现在会识别实际为图片帧列表的 manifest，例如 AVJB `newembed` 的 JPG 序列，并改走内部帧序列播放而不是继续交给标准 Hls.js 视频解码
- **AVJB 播放上下文与降级顺序** - 页面解析后的播放现在会沿用原始来源页的 Referer/Origin，上下文必需时不再走原生 HLS 快捷分支，并且只有在 Hls.js 真正失败后才会降级尝试 JPG 序列播放，使 AVJB 
ewembed 行为更接近原用户脚本
- **页面解析出的短时签名流改为播放时刷新** - 页面解析出的条目现在会保留来源页，并在打开播放器时再次刷新真实流地址，避免 AVJB `newembed` 这类短时签名 m3u8 在真正开始播放前就过期
- **AVJB 的 newembed 隐式回退已扩到 avjb.com** - 当前隐藏网页流解析器现在会同时对 `avjb.cc` 和 `avjb.com` 应用 AVJB 视频/嵌入回退，沿用原用户脚本通过抓取 `/newembed/<videoId>` 提取可播放 m3u8 来绕开网页播放器层的策略
- **现有 M3U 流程内置通用网页流隐式解析** - 当前 M3U 链接加载流程现在会自动区分直链媒体、播放列表文本和普通网页，通过新的通用 Tauri 后端命令提取可播放的 m3u8/mp4/video 源，并直接接入现有 HLS 播放与下载链路，不新增单独 UI
- **Rust 恢复并发与独立调度状态块** - Rust 抓取调度器现在会在连续稳定分段后逐步恢复 `dispatch_limit`，任务状态区也改为用独立状态块展示 Rust 调度指标
- **Rust 慢窗口降块与失败感知 host 降并发** - Rust 抓取调度器现在会在连续慢分段后自动缩小后续 chunk，并区分 request/status/stream/incomplete 失败类型；当 ImgBB 或 Google host 重试过多时会主动降低 dispatch 并发
- **Rust 派发前 chunk 重平衡与调度计数** - Rust 下载器现在会在 worker 出队前按当前自适应 chunk 大小再次拆分过大的 pending 区间，任务状态区也会直接显示 wait/retry/tune/rebalance 计数
- **Rust host 传输配额与实时调度统计** - Rust 抓取下载器现在会按 ImgBB 与 Google 媒体应用不同的分段/块大小上限，记录当前 host 策略，并在任务状态区直接显示 active/pending/EWMA 调度统计
- **Rust 接管分段调度主循环** - 抓取 Rust 下载器现在完整接管分段传输调度，包括 manifest 持久化、pending/in-flight/completed 区间队列、子段续传和最终合并，不再依赖 Python 维护 chunk 分发主循环
- **Rust 剩余区间抢占与慢连接小块化** - Rust 分段下载现在维护共享剩余区间队列，会在 worker 空闲时继续拆分大块，并根据实时吞吐动态缩放后续 chunk，让快连接吃更多剩余区间，慢连接则更快释放
- **Rust 抓取传输实时进度** - Rust `crawler_downloader` 现在会持续输出实时传输进度，Python 会把这些日志实时回灌到抓取任务日志，任务状态区也会汇总 Rust 传输模式、fallback 次数和当前吞吐
- **抓取传输自动预设** - 当 host/range/chunk 调优字段留空时，ImgBB 和 Google 相册 URL 会自动套用更合适的默认并发与分段块大小，减少手动调参
- **抓取 host/range 调优参数** - `Crawler Task` 现在直接提供 host 并发上限、单文件分段 worker 上限、分段块大小覆盖参数，图床异常时无需再手写 extra args
- **Rust 抓取下载器雏形** - 新增独立的 `crawler_downloader` Rust binary，已能读取 crawler 的 `.parts.json` manifest 并输出可续传分段状态，为后续原生下载后端建立真实 CLI 入口
- **抓取下载传输层抽象** - 分段下载和单流下载现在统一走独立的 transport 封装，chunk 调度逻辑可以独立演进，为后续切到 Rust/原生传输后端做准备
- **抓取分段下载 manifest 与 chunk 调度器** - 分段下载现在会在目标文件旁写入 `.parts.json` manifest，用“小块队列 + 有限 worker”方式调度多个 chunk，真正失败时保留 `.part` 子段供下次续传，并根据已完成 chunk 的实时吞吐调整后续 chunk 大小，同时记录 worker/chunk 数量，让续传和多连接行为更接近下载器
- **抓取下载模式开关与分段统计** - `Crawler Task` 现在提供分段下载/单流下载切换，并在任务状态区汇总分段命中、fallback 和子段重试次数
- **默认启用直链媒体分段下载** - 只要爬虫拿到的直链媒体支持 `Accept-Ranges: bytes`，现在都会优先尝试 IDM 风格的 2-4 段并发下载；如果服务端不真正支持分段，则自动回退到单流下载
- **抓取任务状态、日志和图片浏览折叠区** - `Crawler Task` 卡片现在默认展开任务状态，并把实时日志和图片浏览收进独立折叠区，默认界面更干净
- **可收起的爬虫任务参数栏** - 爬虫任务卡片现在保留 URL/输出等基础字段常显，并将高级参数、导入过滤与相关任务选项收进一个可展开/收起的任务参数栏中，默认界面更干净
- **今日抓取与 M3U 维护摘要** - 已将今天完成的爬虫 sidecar、Telegraph 回退、图片浏览器默认态、以及 M3U/HLS 请求链路修复统一收口到 `Unreleased`，便于后续继续维护
- **Telegram 回退抓取任务入口** - Telegraph 原图失效提示和完成弹窗现在会显示 Telegram 回退链接，并提供打开 Telegram、填入当前抓取任务、直接开始抓取三种操作；sidecar 同时透传 `cookies_file` 与登录校验参数
- **爬虫图片浏览默认态、过滤与详情控制** - 图片浏览现在默认更干净地折叠显示，记住详情偏好，支持标题/网页/文件名/落盘路径/已下载等过滤，并提供卡片级与批量的展开/收起控制
- **爬虫任务完成弹窗** - 爬虫任务成功完成后，现在会弹出程序内提示，并可一键打开解析后的输出目录
- **爬虫导入后的 Universal 输出可见性** - 失败重试和导入流程现在会显示并复用真实输出目录，Universal 可直接打开目标目录、显示已保存路径，并支持为直链媒体配置分段下载
- **M3U 自定义请求头输入** - M3U 页面现在支持按来源填写 HTTP 请求头，并会同时用于远程播放列表加载和应用内 HLS 播放
- **程序内下载完成提示弹窗** - Download 与 Universal 队列在每次下载完成后都会显示一个短暂的程序内提示，并可快速打开已保存文件的位置
- **原生 HLS 代理加载器** - 应用内 HLS 播放现在会通过 Tauri 后端统一代理主清单、子清单、媒体分片和解密密钥，而不再只修补顶层 manifest
- **M3U 解析范围扩展** - M3U/M3U8 解析现在会按源播放列表解析相对地址，支持 `#EXT-X-STREAM-INF` 主播放列表，自动去重条目，并保留更多 IPTV 元数据如 `tvg-name`、`tvg-id`、`group-title`
- **M3U/M3U8 流媒体支持** - 爬虫现在会检测并收集 `.m3u` 和 `.m3u8` HLS 播放列表链接。媒体浏览器和查看器通过 hls.js 直接播放 HLS 流。检测到的流可导入 Universal 使用 yt-dlp 下载
- **M3U 工作区与资料管理能力** - 新的 M3U 页面现已整合播放列表加载、频道收藏、加载历史、按目录批量下载，以及基于文件夹的一键收藏管理
- **M3U 播放失败提示和外部播放器** - HLS 或直播流播放失败时（如浏览器不支持 H.265/HEVC 编码），显示错误遮罩层和"用外部播放器打开"及"复制 URL"按钮。Manifest CORS 失败时自动通过 Tauri 后端代理重试
- **HEVC 编解码器检测和安装引导** - 自动检测系统是否安装了 H.265/HEVC 编解码器。未安装且播放失败时，显示蓝色安装卡片并一键跳转 Microsoft Store 安装免费 HEVC Video Extensions
- **M3U/HLS 播放与下载稳定性修复** - M3U/HLS 链路现在会补更完整的播放请求头、正确处理代理 manifest 重写、尊重所选下载目录，并对“只有声音没有画面”这类问题提供更清晰的定位与播放修正
- **爬虫历史文件夹预览与恢复流程** - 历史文件夹现在可自动扫描 `data` 产物、展示更完整的诊断信息、在 sidecar 重启后继续回填最近任务，并直接驱动失败下载重试与预览恢复
- **依赖来源切换（yt-dlp/FFmpeg）** - 在 设置 -> 依赖 中新增来源选择，可在“应用管理”和“系统管理”之间切换
- **切换到系统来源前的安全确认** - 切换 yt-dlp/FFmpeg 到系统来源时新增确认弹窗，避免误操作
- **自定义代理接口类型选择** - 在 AI 设置中新增第三方 OpenAI、OpenAI Responses、NewAPI 网关三种模式，并内联说明后缀与响应差异
- **OpenAI/Proxy 提供商模型发现** - 新增拉取模型动作，可直接从提供商接口获取可用模型并刷新快速选择列表
- **侧栏新增图片抓取入口** - 新增左侧独立入口，可直接打开爬虫抓取控制面板
- **爬虫任务参数扩展** - 新增队列/失败重试文件、模板预设、URL 包含/排除正则、大小与分辨率过滤，复用更多 image_crawler 能力
- **抓取结果媒体浏览面板** - 在爬虫任务下新增内置媒体浏览，支持加载预览、All/Image/GIF/Video/Audio 筛选、缩略网格和打开原链接
- **抓取结果应用内浏览弹窗** - 新增程序内媒体查看弹窗，点击预览卡片即可在应用内打开，并支持按当前筛选结果进行上一项/下一项浏览
- **抓取查看器键盘与图片模式控制** - 在应用内查看器中新增左右方向键、Esc 快捷键，以及图片适应窗口/原始尺寸切换
- **抓取结果重试/导出/分文件夹流程** - 新增当前爬虫媒体失败下载重试、浏览图和下载链接导出、浏览中显示网页原链接，以及导入到 Universal 时按页面自动分子文件夹
- **sidecar 健康诊断字段** - `/health` 新增 build、PID、脚本路径，便于确认当前实际运行的是哪一个 sidecar 实例
- **爬虫模板自动填充** - 选择模板预设（速度优先、高质量、快速预览、严格站点、大规模稳定）后，相关 UI 字段（Workers、超时、延迟、重试、作用域等）会立即自动更新，与 Python `apply_template_defaults()` 逻辑一致
- **爬虫下载速度诊断** - 进度日志每 10 个文件打印下载速率和耗时 (`[PROGRESS] 50/200 (2.5/s, 20s)`)，启动时打印连接池配置 (`[NET] pool_maxsize=12 workers=8 ...`)
- **爬虫下载性能与并发修复** - 通过扩大连接池、修正 RPS 锁竞争、增大流式分块，以及提高默认并发数，批量图片下载不再轻易退化成近似串行执行
- **爬虫任务报错 "Unknown media type(s): m3u, m3u8"** - 从默认媒体类型中移除无效的 `m3u,m3u8`。新增运行时校验，自动过滤 localStorage 缓存中的未知类型
- **Windows 上 Vite 开发服务器 EACCES 权限错误** - 端口 5173 被 Windows Hyper-V/WinNAT 保留。开发服务器改为绑定 `127.0.0.1:9981`，`strictPort: false` 自动降级

- **爬虫信息文件目录轮询已稳定** - 任务轮询现在每轮只计算并写入一次最终的 artifact `data` 路径，不再在输出目录 fallback 与预览推导路径之间来回切换，信息文件目录标签不再闪烁
- **爬虫报告文件现在会迁入媒体文件夹自己的 `data` 目录** - 当一次任务的媒体最终落在单一新建文件夹中时，报告、预览链接、失败列表、检查点和哈希文件会在任务结束后从 `output/data` 移到该媒体文件夹下的 `data` 目录，sidecar 也会跟随新位置查找
- **爬虫信息文件统一写入 `output/data`** - 预览链接、报告、失败列表、检查点、恢复指引与哈希日志现在统一写入独立 `data` 子目录，sidecar 预览收集也会同时检查根目录和 `data` 路径
- **爬虫普通扫描阶段也会导出预览链接** - 正常抓取任务在检测阶段就会刷新 `image_links.txt/csv`，让“加载预览”无需等到下载结束即可读取当前任务媒体
- **系统来源标签按操作系统显示** - 系统来源标签会按平台显示（macOS 显示 Homebrew，Windows 显示 PATH，Linux 显示包管理器）
- **代理路由改为按模式处理** - 代理请求按所选接口类型分流：OpenAI/NewAPI 使用 chat-completions，OpenAI Responses 使用 responses 端点，并带兼容回退路径
- **代理配置规范化保留接口类型** - AI 配置规范化/测试流程会与端点和模型一起持久化 proxy_api_style
- **爬虫文案改为通用媒体范围** - 设置文案明确 sidecar 不仅支持 Google Photos，也支持通用媒体页面
- **sidecar 任务生命周期、恢复与退出行为** - sidecar 现在避免任务锁死，能更稳定地从 `task_running`、瞬时断连与异常轮询中恢复，并在窗口关闭时一起退出，减少残留监听与假占用
- **历史文件夹模式下"重试失败下载"不可用** - 使用文件夹来源模式时，重试失败下载按钮被禁用且无法工作，因为它要求 sidecar 任务 ID。现已支持文件夹模式，直接读取历史文件夹中的 `failed_downloads.txt` 并将失败 URL 重新导入 Universal 队列
- **爬虫开始任务现在优先使用 URL/队列而不是遗留失败重试文件** - 普通开始任务不会再被之前加载的 `retry_failed_from` 路径劫持；只有在 URL 和队列都为空时才进入失败重试模式
- **爬虫失败重试现在会尽量回到原子文件夹** - 失败记录现已保存页面和子目录上下文，重试模式也会从 `download_report.csv` 或唯一媒体文件夹回填旧记录，让失败文件尽可能回到原来的相册子目录
- **抓取浏览空结果会区分运行中状态** - 媒体浏览在任务仍运行时会显示“预览链接尚未生成”的提示，不再过早报“没有可导入链接”
- **直链媒体下载补传页面 Referer** - 从爬虫导入的直链媒体现在会携带来源网页地址作为 `Referer`，提高对防盗链媒体地址的兼容性
- **计划任务通知不再在下载过程中申请权限** - 定时开始/停止/完成通知现在只在系统已授权时发送，避免下载结束后突然弹出系统权限框
